```python
# Наивный RAG: embed -> store -> query -> generate
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_openai import ChatOpenAI

# 1. Загрузка и chunking
docs = TextLoader("data.txt").load()

# 2. Embedding + Store
vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

# 3. Retrieve + Generate
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
chain = RetrievalQA.from_chain_type(llm=ChatOpenAI(), retriever=retriever)
answer = chain.invoke("Что такое RAG?")
```

# 2. ОБЛАСТЬ ЗНАНИЙ

## Часть I. Основы RAG

### Проблема: ограничения LLM
- Знания LLM ограничены датой обучения (knowledge cutoff) -- модель не знает о событиях после этой даты
- Галлюцинации (hallucinations) -- модель уверенно генерирует несуществующие факты, цитаты, ссылки
- Нет доступа к приватным данным -- корпоративные документы, внутренние базы знаний, персональные данные
- Нет контроля источников -- невозможно сказать «отвечай только на основании документа X»
- Fine-tuning не решает проблему полностью: дорого, не масштабируется, не обновляется в реальном времени

### RAG как решение
- Retrieve-Augment-Generate -- трёхэтапный процесс:
  1. Retrieve: найти релевантные документы/фрагменты в базе знаний
  2. Augment: добавить найденные фрагменты в промпт как контекст
  3. Generate: LLM генерирует ответ на основании предоставленного контекста
- Аналогия: RAG -- это «открытая книга на экзамене» для LLM
- Преимущества: актуальность данных, контроль источников, снижение галлюцинаций, attribution (ссылки на источники)

### Наивный RAG
- Простейший пайплайн: embed документы -> сохранить в vector store -> по запросу найти top-k похожих -> вставить в промпт -> сгенерировать ответ
- Архитектура:
```
┌─────────────────────────────────────────────────────────┐
│                    INDEXING (offline)                    │
│                                                         │
│  Documents ──> Chunking ──> Embedding ──> Vector Store  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    QUERYING (online)                     │
│                                                         │
│  Query ──> Embed Query ──> Similarity Search ──> Top-K  │
│                                                         │
│  Top-K + Query ──> Prompt Template ──> LLM ──> Answer   │
└─────────────────────────────────────────────────────────┘
```
- Ограничения наивного RAG: плохой chunking, нет re-ranking, нет обработки сложных запросов, нет фильтрации по метаданным

### Архитектура RAG-пайплайна (компоненты)
- Document Loader -- загрузка из разных источников (PDF, HTML, API, DB)
- Splitter/Chunker -- разбиение на фрагменты оптимального размера
- Embedding Model -- преобразование текста в вектор
- Vector Store -- хранилище векторов с возможностью similarity search
- Retriever -- стратегия поиска (simple, MMR, hybrid, multi-query)
- Re-ranker -- переранжирование результатов по релевантности
- Prompt Template -- шаблон промпта с контекстом и инструкциями
- LLM -- генерация финального ответа
- Post-processor -- проверка ответа, форматирование, attribution

### Когда RAG нужен, а когда нет
- RAG нужен: приватные данные, актуальность критична, нужны ссылки на источники, большой корпус документов
- Fine-tuning лучше: специфический стиль/формат ответа, доменная терминология, постоянный набор знаний
- Long-context LLM лучше: небольшой набор документов (< 100K токенов), один документ за раз, не нужна масштабируемость
- Комбинация: fine-tuned модель + RAG -- часто лучший результат в production
- Таблица сравнения:
```
| Критерий            | RAG           | Fine-tuning   | Long-context  |
|---------------------|---------------|---------------|---------------|
| Обновление данных   | Мгновенно     | Переобучение  | Мгновенно     |
| Масштабируемость    | Миллионы док. | Фиксировано   | Ограничено    |
| Стоимость           | Средняя       | Высокая       | Высокая       |
| Attribution         | Да            | Нет           | Частично      |
| Латентность         | Средняя       | Низкая        | Высокая       |
| Качество на домене  | Высокое       | Очень высокое | Среднее       |
```

## Часть II. Embeddings

### Что такое embeddings
- Вложения (embeddings) -- числовые векторные представления текста в многомерном пространстве
- Семантически похожие тексты расположены близко друг к другу в этом пространстве
- Каждый текст превращается в вектор фиксированной размерности (768, 1024, 1536, 3072 и т.д.)
- Основа similarity search: вместо сравнения строк -- сравнение векторов
- Визуализация:
```
Пространство embeddings (упрощённо до 2D):

  "собака"  *
              \
               * "щенок"
              /
  "кошка"   *
                        * "автомобиль"
                       /
                      * "машина"

Близкие по смыслу ──> близкие в пространстве
```

### Embedding-модели
- **OpenAI**: text-embedding-3-small (1536d), text-embedding-3-large (3072d) -- де-факто стандарт, API-based
- **Cohere**: embed-v3 (1024d) -- хорош для multilingual, поддержка input_type (search_document / search_query)
- **Voyage AI**: voyage-3 (1024d) -- специализация на code и technical docs
- **Открытые модели (self-hosted)**:
  - BAAI/bge-large-en-v1.5 (1024d) -- лидер среди открытых на момент выхода
  - nomic-embed-text (768d) -- длинный контекст (8192 tokens), компактный
  - mixedbread-ai/mxbai-embed-large (1024d) -- высокая производительность
  - Snowflake/snowflake-arctic-embed (варианты от 384d до 1024d) -- отличное соотношение размер/качество
- Выбор: API (OpenAI, Cohere) для старта, self-hosted (BGE, Nomic) для контроля и стоимости

### Размерность и trade-offs
- Больше размерность -- больше информации, но: медленнее поиск, больше хранилища, выше стоимость
- Matryoshka embeddings: модели (text-embedding-3) позволяют обрезать вектор до нужной размерности без потери качества
- Практический выбор: 768-1024d -- оптимум для большинства задач; 256-384d -- для edge/mobile; 1536-3072d -- когда качество критично
- Квантизация: float32 -> float16 -> int8 -> binary -- каждый шаг уменьшает хранилище и ускоряет поиск, но снижает точность

### Sentence vs Document embeddings
- Sentence embeddings: один вектор на предложение/абзац (sentence-transformers) -- лучше для точного retrieval
- Document embeddings: один вектор на весь документ -- теряет детали, подходит для грубой классификации
- На практике: chunking + sentence-level embeddings -- стандарт для RAG
- Late chunking: embed весь документ, затем извлечь embeddings для чанков с сохранением контекста документа

### Multimodal embeddings
- Текст + изображения в одном пространстве: CLIP (OpenAI), SigLIP (Google), Nomic Embed Vision
- Применение: поиск изображений по текстовому запросу, RAG по презентациям и PDF с картинками
- ColPali / ColQwen: визуальные модели, которые обрабатывают страницу документа как изображение, минуя OCR и парсинг
- Trade-off: multimodal модели обычно хуже чисто текстовых на текстовых задачах

### Бенчмарки
- **MTEB (Massive Text Embedding Benchmark)** -- стандартный бенчмарк для embedding-моделей
- Задачи MTEB: classification, clustering, pair classification, reranking, retrieval, STS, summarization
- MTEB Leaderboard (Hugging Face) -- рейтинг моделей, обновляется регулярно
- Важно: смотри на retrieval-подмножество, а не общий score -- для RAG это ключевая метрика
- Бенчмарки на своих данных: общий рейтинг не гарантирует качество на конкретном домене. Всегда тестируй на своих данных

### Выбор модели под задачу
- Чеклист выбора:
  1. Язык данных (multilingual? только английский? русский?)
  2. Тип контента (код? юридические документы? чат?)
  3. Максимальная длина текста (128 tokens? 8192 tokens?)
  4. Ограничения: бюджет, latency, self-hosted vs API
  5. Privacy: можно ли отправлять данные через API?
- Рекомендация: начинай с OpenAI text-embedding-3-small, замеряй качество на своих данных, затем пробуй альтернативы

## Часть III. Vector Databases

### Зачем нужны
- Similarity search at scale: найти ближайшие векторы среди миллионов за миллисекунды
- Обычные базы данных (PostgreSQL, MongoDB) не оптимизированы для поиска в многомерном пространстве
- Vector databases предоставляют: ANN-индексы, фильтрацию по метаданным, управление коллекциями, масштабирование

### Алгоритмы поиска ближайших соседей

#### Brute-force (точный поиск)
- Сравнение запроса со ВСЕМИ векторами в базе
- O(n * d) где n -- количество векторов, d -- размерность
- Идеальная точность, но не масштабируется (> 100K документов -- уже медленно)
- Используй для: тестирования, маленьких коллекций, ground truth при оценке ANN

#### HNSW (Hierarchical Navigable Small World)
- Граф с несколькими уровнями: верхний уровень -- грубая навигация, нижний -- точная
- Быстрый поиск: O(log n), высокая точность (recall > 95%)
- Trade-off: потребление памяти (хранит граф в RAM)
- Визуализация:
```
Level 2:  A ──────────── D
          |               |
Level 1:  A ── B ──── D ── E
          |    |      |    |
Level 0:  A-B--C--D-E--F--G--H   (все точки)

Поиск: начинаем с верхнего уровня, спускаемся к точным соседям
```
- Параметры: M (число связей), efConstruction (качество построения), efSearch (качество поиска)

#### IVF (Inverted File Index)
- Кластеризация векторов (k-means), поиск только в ближайших кластерах
- Быстрее HNSW по памяти, но ниже recall при малом nprobe
- Параметры: nlist (число кластеров), nprobe (сколько кластеров проверять)
- Подходит для: очень большие коллекции (> 10M), GPU-ускорение (FAISS)

#### PQ (Product Quantization)
- Сжатие векторов: разбиение на подвекторы, квантизация каждого
- Сильно уменьшает потребление памяти (8-32x)
- Потеря точности: подходит как первый этап + reranking точными векторами
- Часто комбинируют: IVF-PQ, HNSW-PQ

### Решения

#### Chroma
- Простой, встроенный (in-process), идеален для прототипов и dev
- Python-native, минимум конфигурации
- Ограничения: не для production на больших данных, нет кластеризации

#### Pinecone
- Fully managed (SaaS), serverless или pod-based
- Быстрый старт, не нужно управлять инфрой
- Ограничения: vendor lock-in, стоимость растёт с масштабом, данные хранятся у провайдера

#### Weaviate
- Open-source, self-hosted или cloud
- Встроенная векторизация (модуль), GraphQL API, hybrid search
- Хорош для: мультимодальный поиск, сложные схемы данных

#### Qdrant
- Open-source, написан на Rust (высокая производительность)
- Мощная фильтрация по payload (метаданным), gRPC API
- Хорош для: production self-hosted, сложные фильтры, многоязычные данные

#### pgvector
- Расширение для PostgreSQL -- embeddings в знакомой СУБД
- Плюс: не нужна отдельная инфраструктура, SQL, транзакции, joins
- Минус: медленнее специализированных решений на больших объёмах
- Хорош для: команды с PostgreSQL-стеком, < 5M векторов

#### Milvus
- Open-source, горизонтальное масштабирование, GPU-поддержка
- Для больших масштабов: миллиарды векторов
- Сложнее в настройке и эксплуатации

### Сравнение решений
```
| Решение  | Тип         | Масштаб   | Hybrid | Фильтры | Старт    |
|----------|-------------|-----------|--------|---------|----------|
| Chroma   | Embedded    | < 1M     | Нет    | Базовые | Минуты   |
| Pinecone | Managed     | < 1B     | Да     | Да      | Минуты   |
| Weaviate | Self/Cloud  | < 100M   | Да     | Да      | Часы     |
| Qdrant   | Self/Cloud  | < 100M   | Да     | Мощные  | Часы     |
| pgvector | Extension   | < 5M     | Нет*   | SQL     | Минуты   |
| Milvus   | Self-hosted | < 10B    | Да     | Да      | Дни      |
```

### Индексирование и оптимизация
- Выбор типа индекса под объём и требования по latency/recall
- Настройка параметров HNSW: M=16 (стандарт), efConstruction=200, efSearch=100
- Batch indexing vs online: массовая загрузка быстрее, но требует downtime
- Партиционирование: по tenant, по дате, по типу документа

### Metadata filtering
- Фильтрация до или после similarity search
- Pre-filter: сужает пространство поиска (быстрее, но может пропустить релевантные результаты при жёстких фильтрах)
- Post-filter: сначала поиск, потом фильтрация (точнее, но медленнее)
- Примеры метаданных: source, date, author, category, language, access_level

### Hybrid search
- Комбинация vector search (семантика) + keyword search (точное совпадение, BM25)
- Почему нужен: vector search плохо работает с именами, ID, точными терминами
- Архитектура:
```
Query ──┬──> Vector Search ──> Results_v (scored)
        │
        └──> Keyword Search ──> Results_k (scored)
                                    │
                        ┌───────────┘
                        v
                  Reciprocal Rank Fusion (RRF)
                        │
                        v
                  Merged Results (re-ranked)
```
- RRF (Reciprocal Rank Fusion) -- стандартный способ объединения рангов из разных источников
- Weaviate, Qdrant, Pinecone поддерживают hybrid search нативно

## Часть IV. Chunking

### Зачем нужен chunking
- LLM имеет ограниченное контекстное окно -- нельзя вставить весь документ
- Маленькие чанки -- точнее retrieval, но теряют контекст
- Большие чанки -- больше контекста, но ниже precision, дороже по токенам
- Задача: найти оптимальный баланс между точностью retrieval и достаточным контекстом для генерации

### Стратегии chunking

#### Fixed-size (фиксированный размер)
- Разбиение по символам/токенам: каждые N символов -- новый чанк
- Просто, предсказуемо, но рвёт предложения и абзацы посередине
- Используй только как baseline для сравнения

#### Sentence-based (по предложениям)
- Разбиение по границам предложений (nltk, spaCy, regex)
- Группировка: N предложений в чанк
- Лучше fixed-size, но не учитывает структуру документа

#### Recursive (рекурсивный)
- RecursiveCharacterTextSplitter (LangChain) -- де-факто стандарт
- Пытается разбить по: \n\n -> \n -> . -> пробел -> символ
- Сохраняет структуру: абзацы > предложения > слова
- Рекомендация по умолчанию для большинства текстов

#### Semantic (семантический)
- Группировка предложений на основе similarity их embeddings
- Если cosine similarity между соседними предложениями падает ниже порога -- разрез
- Плюс: чанки содержат семантически связанную информацию
- Минус: медленнее (нужно embed каждое предложение), непредсказуемый размер

### Overlap (перекрытие)
- Overlap -- дублирование конца предыдущего чанка в начале следующего
- Зачем: сохранение контекста на границах чанков
- Типичные значения: 10-20% от размера чанка (chunk_size=512, overlap=50-100)
- Слишком большой overlap: дублирование данных, увеличение хранилища, шум в retrieval
- Слишком маленький: потеря контекста на границах

### Document-aware chunking
- Использование структуры документа: заголовки (H1, H2, H3), секции, списки
- MarkdownHeaderTextSplitter (LangChain): разбивает по заголовкам, сохраняет иерархию метаданных
- HTMLHeaderTextSplitter: аналогично для HTML
- Добавление метаданных: section_title, chapter, page_number

### Code chunking
- Код нельзя разбивать по символам/строкам -- ломается синтаксис
- AST-based chunking: разбиение по функциям, классам, модулям (tree-sitter)
- Формат: каждый чанк = функция/класс + docstring + сигнатура
- LangChain: RecursiveCharacterTextSplitter с language-specific разделителями

### Chunking для разных типов документов
- **PDF**: unstructured, PyMuPDF, pdfplumber. Проблемы: таблицы, многоколоночная вёрстка, сканы (OCR)
- **HTML**: парсинг + очистка (BeautifulSoup), chunking по тегам
- **Code**: AST-based (tree-sitter), по функциям/классам
- **Tables**: сохранение таблицы целиком или построчно с заголовками. Не разрывать строки таблицы
- **Markdown**: по заголовкам (MarkdownHeaderTextSplitter)

### Размер чанка: trade-offs
```
| Размер чанка     | Retrieval precision | Контекст | Стоимость LLM | Рекомендация          |
|------------------|---------------------|----------|---------------|------------------------|
| 128-256 tokens   | Высокая             | Мало     | Низкая        | Точечные факты, FAQ    |
| 512-1024 tokens  | Средняя             | Средне   | Средняя       | Общий текст (стандарт) |
| 1024-2048 tokens | Низкая              | Много    | Высокая       | Связные тексты, отчёты |
```
- Эмпирическое правило: начинай с 512-1024 tokens, overlap 10-20%, замеряй метрики, оптимизируй

## Часть V. Retrieval-стратегии

### Similarity search (поиск по сходству)
- Метрики расстояния:
  - **Cosine similarity**: нормализованное скалярное произведение, стандарт для текстовых embeddings (0..1)
  - **Dot product**: без нормализации, быстрее, но чувствителен к длине вектора
  - **Euclidean distance (L2)**: расстояние в пространстве, менее интуитивен для текста
- Выбор: cosine similarity -- стандарт для RAG. Dot product -- если модель обучена на нём (проверяй документацию модели)

### MMR (Maximum Marginal Relevance)
- Проблема: top-k результатов могут быть слишком похожими (дублирование информации)
- MMR: баланс между релевантностью к запросу и разнообразием результатов
- Формула: MMR = argmax[ lambda * sim(d, q) - (1 - lambda) * max(sim(d, d_selected)) ]
- lambda = 1.0: только релевантность (обычный top-k); lambda = 0.0: только разнообразие
- Рекомендация: lambda = 0.5-0.7 для RAG -- хороший баланс

### Re-ranking (переранжирование)
- Двухэтапная схема: быстрый retrieval (bi-encoder, ANN) -> точный re-ranking (cross-encoder)
- Bi-encoder: embed query и document отдельно, сравнить -> быстро, но менее точно
- Cross-encoder: query + document вместе через модель -> точно, но медленно (O(n))
```
Query ──> Bi-Encoder ──> Top-100 ──> Cross-Encoder ──> Top-5 (re-ranked)
          (быстро)                    (точно)
```
- Инструменты:
  - **Cohere Rerank**: API, высокое качество, multilingual
  - **cross-encoder/ms-marco-MiniLM-L-6-v2**: open-source, можно запускать локально
  - **FlashRank**: быстрый, лёгкий re-ranker для self-hosted
  - **RankGPT / LLM-based reranking**: использование LLM для ранжирования (дорого, но качественно)

### HyDE (Hypothetical Document Embeddings)
- Проблема: запрос пользователя и релевантный документ могут быть стилистически далеки
- Идея: LLM генерирует «гипотетический ответ» на запрос, затем ищем по embedding этого ответа
- Пайплайн: Query -> LLM (generate hypothetical doc) -> Embed -> Search
- Плюс: ближе к стилю документов в базе
- Минус: дополнительный LLM-вызов (latency, cost), может усилить hallucinations на плохих запросах

### Query expansion и decomposition
- **Query expansion**: добавление синонимов, переформулировка запроса
  - LLM генерирует 3-5 вариантов запроса, результаты объединяются
- **Query decomposition**: разбиение сложного запроса на под-вопросы
  - «Сравни RAG и fine-tuning для чатбота» -> «Что такое RAG?» + «Что такое fine-tuning?» + «Преимущества RAG для чатботов» + «Преимущества fine-tuning для чатботов»
- Multi-query retrieval: несколько запросов параллельно, результаты объединяются через union или RRF

### Contextual retrieval (подход Anthropic)
- Проблема: чанки теряют контекст документа после разбиения
- Решение: для каждого чанка LLM генерирует краткое описание контекста (contextual header)
- Формат: «Этот чанк из раздела X документа Y, описывает Z. [оригинальный текст чанка]»
- Contextual header добавляется перед текстом чанка перед embedding
- Результат: +15-20% к retrieval recall на бенчмарках Anthropic

### Parent Document Retriever
- Проблема: маленькие чанки -- хороший retrieval, но мало контекста для LLM
- Решение: индексировать маленькие чанки (child), а возвращать большие (parent)
- Маленький чанк (256 tokens) для точного поиска, большой (2048 tokens) для контекста
```
Parent chunk (2048 tokens): [=============================================]
Child chunks (256 tokens):  [=====][=====][=====][=====][=====][=====]
                                     ^
                              найден по запросу
                                     |
                              возвращаем parent
```

### Multi-query retrieval
- LLM генерирует несколько вариантов запроса с разных точек зрения
- Каждый вариант -- отдельный retrieval
- Результаты объединяются (union unique)
- Плюс: покрывает разные формулировки и аспекты запроса
- Минус: N * latency retrieval + LLM-вызов для генерации вариантов

## Часть VI. Advanced RAG

### Agentic RAG
- Агент решает: нужно ли вообще обращаться к базе знаний, и к какой именно
- Инструменты агента: vector_search, sql_query, web_search, calculator
- Агент может делать несколько retrieval-итераций, уточняя запрос
- Архитектура:
```
User Query ──> Agent (LLM) ──┬──> Tool: Vector Search ──> Results
                              │──> Tool: SQL Query ──> Results
                              │──> Tool: Web Search ──> Results
                              └──> Synthesize Answer
```
- Реализация: LangChain Agents, LlamaIndex Agents, CrewAI

### Corrective RAG (CRAG)
- После retrieval: LLM оценивает релевантность найденных документов
- Если документы нерелевантны -- запускается альтернативный поиск (web search, другой индекс)
- Если частично релевантны -- выделяются только полезные фрагменты
- Пайплайн:
```
Query ──> Retrieve ──> Evaluate Relevance ──┬──> Correct: use docs
                                             ├──> Incorrect: web search
                                             └──> Ambiguous: refine + retry
```

### Self-RAG
- Модель сама решает: (a) нужен ли retrieval, (b) какие документы использовать, (c) насколько ответ поддержан источниками
- Reflection tokens: [Retrieve], [IsRel], [IsSup], [IsUse]
- Модель обучена генерировать эти токены как часть ответа
- Более сложный, но более контролируемый RAG

### Graph RAG (knowledge graphs + vector search)
- Проблема: vector search не улавливает связи между сущностями
- Решение: построить knowledge graph из документов, использовать для навигации и ответов
- Этапы: extract entities -> build graph -> community detection -> summarize communities -> retrieve + traverse
- Microsoft Graph RAG: LLM извлекает сущности и связи, строит граф, ищет по нему
- Комбинация: vector search для документов + graph traversal для связей
- Когда нужен: сложные вопросы о связях («кто из партнёров компании X работал с Y?»), многоэтапные рассуждения

### Multi-modal RAG
- RAG по документам с изображениями, таблицами, графиками
- Подходы:
  1. Multimodal embeddings (CLIP/SigLIP): embed и текст и изображения в одно пространство
  2. Vision LLM для описания изображений: GPT-4V / Claude описывает картинку -> текст -> embed
  3. ColPali: embed страницу документа как изображение, минуя парсинг
- Таблицы: преобразование в markdown/HTML перед chunking или обработка vision-моделью

### Streaming RAG
- Генерация ответа начинается до завершения retrieval? Нет -- retrieval обязателен до генерации
- Streaming = потоковый вывод ответа LLM (token by token) после retrieval
- UX: показать «Ищу информацию...» -> показать источники -> потоковый вывод ответа
- Реализация: Server-Sent Events (SSE), WebSocket

### RAG с structured data
- SQL RAG: Text-to-SQL -> выполнение запроса -> ответ на основе результата
- API RAG: агент вызывает API для получения данных (погода, курсы, статусы)
- Комбинация: unstructured (документы) + structured (SQL, API) в одном пайплайне
- NL2SQL: опасность SQL-инъекций! Валидация, песочница, read-only доступ

## Часть VII. Evaluation

### Метрики retrieval

#### Precision@k
- Доля релевантных документов среди top-k результатов
- Precision@5 = (число релевантных в top-5) / 5
- Высокий precision -- мало мусора в результатах

#### Recall@k
- Доля найденных релевантных документов из всех релевантных в коллекции
- Recall@10 = (релевантные в top-10) / (все релевантные в коллекции)
- Высокий recall -- не пропускаем важные документы

#### MRR (Mean Reciprocal Rank)
- Средний обратный ранг первого релевантного результата
- MRR = 1/N * sum(1/rank_i)
- Показывает: как быстро пользователь увидит первый правильный результат

#### NDCG (Normalized Discounted Cumulative Gain)
- Учитывает позицию и степень релевантности (graduated relevance)
- Релевантный документ на 1 позиции ценнее, чем на 10-й
- Стандарт для оценки поисковых систем

### Faithfulness и relevance (метрики генерации)
- **Faithfulness** (верность): ответ основан на предоставленном контексте, а не выдуман
- **Relevance** (релевантность): ответ действительно отвечает на вопрос
- **Context relevance**: найденные документы релевантны запросу
- Эти метрики оценивают весь RAG-пайплайн, не только retrieval

### RAGAS framework
- Автоматическая оценка RAG-систем с использованием LLM
- Метрики RAGAS:
  - **Faithfulness**: извлечение утверждений из ответа, проверка каждого по контексту
  - **Answer relevance**: генерация вопросов из ответа, сравнение с исходным вопросом
  - **Context precision**: доля полезных чанков среди извлечённых
  - **Context recall**: покрытие ground truth ответа извлечённым контекстом
- Установка и использование:
```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

result = evaluate(
    dataset,  # questions + ground_truth + contexts + answers
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)
print(result)  # {'faithfulness': 0.87, 'answer_relevancy': 0.92, ...}
```

### LLM-as-judge для RAG
- LLM оценивает качество ответа по критериям (prompt-based evaluation)
- Критерии: полнота, точность, ссылки на источники, отсутствие галлюцинаций
- Проблемы: bias LLM-судьи, стоимость, нестабильность оценок
- Рекомендация: использовать как дополнение к RAGAS, не как единственную метрику

### End-to-end evaluation
- Полная оценка: от вопроса до финального ответа
- Ground truth dataset: набор (вопрос, эталонный ответ, релевантные документы)
- Создание ground truth: ручная разметка (золотой стандарт) или LLM-генерация + ручная проверка
- Минимальный тестовый набор: 50-100 вопросов с аннотациями

### A/B тестирование RAG-пайплайнов
- Сравнение двух конфигураций на одном наборе запросов
- Что варьировать: chunk_size, embedding model, retriever type, re-ranker, prompt template
- Автоматизация: скрипт прогоняет обе конфигурации на тестовом наборе, сравнивает метрики
- Статистическая значимость: paired t-test или bootstrap для оценки разницы
- Инструменты: LangSmith, Weights & Biases, custom evaluation scripts

## Часть VIII. Production

### Caching стратегии
- **Semantic cache**: кеширование ответов по similarity запроса (GPTCache)
  - Если новый запрос семантически похож на кешированный -> вернуть кешированный ответ
  - Порог similarity: 0.95+ для точных совпадений
- **Embedding cache**: кеширование вычисленных embeddings (не пересчитывать для известных текстов)
- **LLM response cache**: кеширование ответов LLM для идентичных промптов
- Invalidation: по TTL, при обновлении документов, ручной сброс

### Инкрементальное обновление индекса
- Проблема: переиндексация всего корпуса дорога и медленна
- Решение: upsert (обновление/добавление отдельных документов), delete по ID
- Change detection: хеширование содержимого, отслеживание mtime, webhook при обновлении
- Версионирование: metadata с version/timestamp, фильтрация по актуальности

### Мониторинг качества retrieval
- Метрики для мониторинга:
  - Средний retrieval score (similarity) -- падение = деградация модели или данных
  - Процент запросов без результатов (no-hit rate)
  - Latency (p50, p95, p99) для retrieval и generation
  - Feedback loop: пользовательские оценки ответов (thumbs up/down)
- Инструменты: LangSmith, Langfuse, Phoenix (Arize), custom dashboards
- Алерты: если retrieval score < threshold, no-hit rate > X%, latency > Y ms

### Cost optimization
- Основные затраты: embedding API, LLM API, хранилище векторов, compute
- Оптимизация embedding: batch processing, кеширование, self-hosted модели для больших объёмов
- Оптимизация LLM: короткие промпты, меньше контекста (лучше retrieval -> меньше чанков), маленькие модели для простых запросов
- Оптимизация хранилища: квантизация векторов, удаление устаревших документов, партиционирование
- Router: маршрутизация простых запросов на дешёвые модели, сложных -- на мощные

### Scaling
- Горизонтальное масштабирование vector store: шардирование, репликация
- Масштабирование retrieval: кеш, async retrieval, параллельные запросы к нескольким индексам
- Масштабирование LLM: rate limiting, очереди, fallback на альтернативные модели
- Multi-tenant: изоляция данных по namespace/collection, фильтрация по tenant_id
- Архитектура production RAG:
```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│   API GW    │────>│  RAG Service │────>│  LLM Gateway │
│  (auth,     │     │  (retrieval, │     │  (routing,   │
│   rate lim) │     │   augment)   │     │   fallback)  │
└─────────────┘     └──────┬───────┘     └──────────────┘
                           │
              ┌────────────┼────────────┐
              v            v            v
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Vector   │ │ Cache    │ │ Metadata │
        │ Store    │ │ (Redis)  │ │ Store    │
        └──────────┘ └──────────┘ └──────────┘
```

### Benchmark challenge

Формат:
```
**Сценарий:** RAG-система для юридических документов.
Текущие метрики: Precision@5 = 0.6, Recall@10 = 0.75, MRR = 0.5

**Вариант A:** Увеличить chunk_size с 512 до 1024 tokens, добавить overlap 200.
**Вариант B:** Оставить chunk_size=512, добавить Cohere re-ranker после top-20 retrieval.
```

# 8. МЕЖДИСЦИПЛИНАРНЫЕ СВЯЗИ

## RAG и NLP
- Токенизация, BM25, TF-IDF -- классические методы информационного поиска, основа hybrid search
- Transformer-архитектура -- основа как embedding-моделей, так и LLM
- Named Entity Recognition (NER) -- используется в Graph RAG для извлечения сущностей

## RAG и базы данных
- Индексы в RDBMS (B-tree) vs ANN-индексы (HNSW, IVF) -- разные задачи, разные структуры данных
- SQL для structured data RAG, vector search для unstructured
- pgvector -- мост между реляционным миром и векторным поиском

## RAG и MLOps
- Мониторинг drift: изменение распределения запросов, деградация embedding-модели
- CI/CD для RAG: автоматические тесты на тестовом датасете при изменении пайплайна
- Feature store аналогии: vector store как «embedding store»

## RAG и безопасность
- Prompt injection через документы в базе знаний
- Data poisoning: вставка вредоносного контента в индексируемые документы
- Access control: пользователь не должен видеть документы, к которым нет доступа (metadata filtering по access_level)
- PII в чанках: необходимость маскирования персональных данных перед индексацией

## RAG и UX
- Streaming ответов -- восприятие скорости
- Attribution (ссылки на источники) -- доверие пользователя
- «Я не знаю» лучше чем hallucination -- fallback-стратегия
- Feedback loop: thumbs up/down -> улучшение retrieval

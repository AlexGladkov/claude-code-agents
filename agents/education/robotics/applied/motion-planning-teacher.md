# 2. ОБЛАСТЬ ЗНАНИЙ

## Часть I. Конфигурационное пространство (C-space)

### Что такое C-space
- Конфигурация q: минимальный набор параметров, полностью описывающий положение робота
- Примеры конфигурации:
  - Точечный робот в 2D: q = (x, y) — 2 DOF
  - Мобильный робот: q = (x, y, θ) — 3 DOF (SE2)
  - 6-DOF манипулятор: q = (θ1, θ2, θ3, θ4, θ5, θ6) — 6 DOF
  - Гуманоид: q ∈ R^50 и выше
- C-space vs рабочее пространство (workspace): планирование всегда в C-space

### C-obstacles
- C-obstacle: область конфигураций, при которых робот пересекается с препятствием
- Конструкция: C-obstacle = {q : A(q) ∩ B ≠ ∅} где A(q) — форма робота, B — препятствие
- Minkowski sum: для вычисления C-obstacles сферического робота: растяжение препятствия на радиус робота
- Для манипуляторов: C-obstacles имеют сложную форму — аналитически редко вычислимо

### Свободное пространство (C-free)
- C-free = C \ C-obstacle: все допустимые конфигурации
- Топология C-free: может быть несвязным, иметь узкие проходы (narrow passages)
- Полнота алгоритма (completeness): найдёт ли путь если он существует?
- Вероятностная полнота (probabilistic completeness): вероятность нахождения → 1 при числе итераций → ∞
- Размерность проклятия (curse of dimensionality): объём C-space растёт экспоненциально с DOF

### Движение в C-space
- Local planner: проверка коллизии между двумя конфигурациями (дискретная проверка)
- Collision checking: `fcl` (Flexible Collision Library), `bullet3` — стандарты
- Sampling density: насколько дискретно проверять путь между конфигурациями
- Configuration distance: метрика в C-space (L2-норма для суставных углов с весами)

## Часть II. Sampling-based планирование: RRT, RRT*, PRM

### Вероятностная дорожная карта (PRM)
- Идея: заранее сэмплируй C-free, соединяй ближайших соседей → offline граф
- Алгоритм:
  1. Фаза обучения: N случайных конфигураций в C-free + соединение k-nearest neighbors
  2. Фаза запроса: добавить start и goal к графу, найти путь (A*)
- Multi-query: один roadmap для множества запросов — эффективно для статичной среды
- Ограничение: плохо работает в узких проходах, требует много конфигураций
- Lazy PRM: откладывай collision checking до запроса — быстрее для редких запросов

### RRT — Rapidly-exploring Random Tree
- Идея: итеративно строить дерево из start, направляя к случайным точкам C-space

```
Алгоритм RRT:
T.init(q_start)
for i = 1 to N:
    q_rand = SampleFree()         // случайная конфигурация
    q_near = NearestNeighbor(T, q_rand)  // ближайшая вершина в дереве
    q_new = Steer(q_near, q_rand, step_size)  // шаг в направлении q_rand
    if CollisionFree(q_near, q_new):
        T.add_vertex(q_new)
        T.add_edge(q_near, q_new)
        if Reached(q_new, q_goal):
            return Path(T, q_start, q_new)
```

- Goal biasing: с вероятностью p_goal сэмплировать цель — ускоряет сходимость
- Вероятностная полнота: да, при p_goal < 1 и достаточно итераций
- Оптимальность: нет — первый найденный путь, не лучший
- Сложность: O(n log n) при эффективном поиске ближайших соседей (kd-tree)

### RRT* — асимптотически оптимальный RRT
- Ключевые добавления к RRT:
  1. **Rewiring:** при добавлении q_new проверяем: не лучше ли подключить через q_new соседей в радиусе r?
  2. **Radius:** r = γ * (log(n)/n)^(1/d) — гарантирует асимптотическую оптимальность
- Асимптотическая оптимальность: при n → ∞ стоимость найденного пути → оптимальной
- Практика: уже через несколько тысяч итераций путь значительно лучше RRT
- Недостаток: медленнее RRT в каждой итерации из-за rewiring

### Bidirectional RRT (RRT-Connect)
- Два дерева: от start и от goal; чередующееся расширение
- CONNECT: extend не на один step, а пока не достигнем цели или препятствия
- Значительно быстрее однонаправленного RRT для простых сред
- Реализация: `ompl::geometric::RRTConnect`

### Informed RRT*
- После нахождения первого пути: сэмплируй только из эллипсоида вокруг start-goal
- Эллипсоид: все точки, путь через которые лучше текущего лучшего
- Резко ускоряет улучшение пути по сравнению с RRT*

### Продвинутые варианты
- KPIECE, EST, SBL: альтернативные стратегии расширения из OMPL
- CHOMP, STOMP: траекторная оптимизация (gradient-based, stochastic)
- Lazy collision checking: откладывай дорогую проверку коллизии

## Часть III. Graph-based планирование: A*, D*, Theta*

### Дискретизация C-space
- Grid map: 2D/3D сетка, каждая ячейка — occupied/free
- Разрешение: чем мельче сетка — точнее путь, дороже поиск
- 4-connectivity vs 8-connectivity vs 26-connectivity (3D)
- Weighted graph: стоимость перехода зависит от terrain (склон, мягкий грунт)

### A* — эвристический поиск кратчайшего пути
- f(n) = g(n) + h(n): стоимость до n + эвристическая оценка до цели
- Эвристика h: должна быть допустимой (admissible): h(n) ≤ h*(n)
  - Для 4-connectivity: Manhattan distance
  - Для 8-connectivity: Chebyshev distance (Diagonal distance)
  - Для евклидова пространства: Euclidean distance
- Оптимальность: да, при допустимой эвристике
- Полнота: да, при конечном графе
- Сложность: O(b^d) в худшем случае; с хорошей эвристикой — значительно лучше

```
A* псевдокод:
OPEN = {start}; g[start] = 0
while OPEN не пустое:
    n = argmin f(x) в OPEN  // приоритетная очередь (min-heap)
    if n == goal: return reconstruct_path()
    OPEN.remove(n); CLOSED.add(n)
    for each neighbor x of n:
        if x в CLOSED: continue
        tentative_g = g[n] + cost(n, x)
        if x ∉ OPEN or tentative_g < g[x]:
            g[x] = tentative_g
            f[x] = g[x] + h(x)
            parent[x] = n
            OPEN.add(x)
```

### D* и D* Lite — переплановщик
- Проблема A*: при изменении карты нужно перепланировать с нуля
- D* (Dynamic A*): перепланирует только изменённые части графа — эффективно
- D* Lite (Koenig & Likhachev 2002): упрощённая версия D*, используется в практике
  - Планирует от goal к start (для эффективного перепланирования)
  - Применение: Mars Rover (NASA использовал D* для планирования Spirit/Opportunity)

### Theta* — any-angle planning
- Проблема A* на сетке: пути ограничены диагоналями сетки, выглядят «ступенчато»
- Theta*: разрешает переходы между любыми вершинами если line-of-sight свободен
  - Line-of-sight проверка: Bresenham's line algorithm
- Пути ближе к евклидовому оптимуму без post-processing сглаживания
- Lazy Theta*: откладывает line-of-sight проверку — быстрее при высокой стоимости коллизии

### Dijkstra's Algorithm
- A* с h(n) = 0 — исследует волнами от start, гарантированно оптимален
- Применение: когда нет хорошей эвристики или граф взвешен нетривиально
- Алгоритм Беллмана-Форда: для графов с отрицательными весами

### Сравнение алгоритмов graph-based

```
| Алгоритм   | Оптимален | Динамическая среда | Any-angle | Сложность  |
|------------|-----------|-------------------|-----------|------------|
| Dijkstra   | Да        | Нет (перепланир.) | Нет       | O(V log V) |
| A*         | Да        | Нет (перепланир.) | Нет       | O(b^d)     |
| D* Lite    | Да        | Да (эфф.)         | Нет       | O(k log k) |
| Theta*     | ~Да       | Нет               | Да        | O(b^d)     |
| Lazy Theta*| ~Да       | Нет               | Да        | Быстрее    |
```

## Часть IV. Потенциальные поля и реактивное управление

### Метод потенциальных полей (Artificial Potential Fields)
- Идея: робот движется как частица в силовом поле
  - Аттрактивный потенциал: U_att = 1/2 * ξ * d²(q, q_goal) — притяжение к цели
  - Репульсивный потенциал: U_rep = η/2 * (1/d - 1/d0)² если d < d0, иначе 0
  - Суммарная сила: F = -∇U = F_att + F_rep
- Простота реализации: вычисляй градиент → двигай робота
- Преимущество: плавные траектории, реакция в реальном времени
- Критическая проблема: локальные минимумы — «ловушки», из которых робот не выйдет
  - U-образное препятствие полностью блокирует традиционный метод

### Навигационные функции (Navigation Functions)
- Koditschek & Rimon (1990): построение потенциала без локальных минимумов
- Гарантия: один минимум = цель, при условии «звёздной» (star-shaped) топологии C-free
- Ограничение: требует аналитического описания препятствий — не для общего случая

### Vector Field Histogram (VFH)
- VFH: алгоритм реактивного управления мобильного робота
  - Polar histogram: распределение препятствий по направлениям вокруг робота
  - Valley selection: выбор направления с наименьшей плотностью препятствий
- VFH+: учитывает кинематику (минимальный радиус поворота)
- VFH*: комбинация с A* для глобального направления

### Dynamic Window Approach (DWA)
- Ключевая идея: ограничь пространство управляющих воздействий динамикой робота
  - Dynamic window: достижимые (v, ω) за δt с учётом ускорений
  - Оценочная функция: цель + clearance + velocity — выбери лучшую (v, ω)
- Используется в Nav2 (DWB controller): стандарт для дифференциальных роботов
- Ограничение: жадный алгоритм — нет глобального оптимума, возможны тупики

### TEB (Timed Elastic Band)
- Резиновая лента во времени: траектория как последовательность поз с временными метками
- Оптимизируется одновременно: геометрия + скорости + ограничения (кинематика, препятствия)
- Поддерживает Ackermann-steering, holonomic, differential drive
- Применение: Nav2 как альтернатива DWB для неголономных роботов

## Часть V. MoveIt для манипуляторов

### Архитектура MoveIt (обзор с практической точки зрения)
- MoveGroupInterface: Python/C++ API — главный инструмент для разработчика
- Planning Pipeline: `planner_plugin` → `request_adapter` → `response_adapter`
- Planning Scene: виртуальный мир с роботом и препятствиями
- Kinematic Solvers: IK plugin для решения обратной кинематики

### Прямая и обратная кинематика
- Forward Kinematics (FK): θ1...θn → pose end-effector (аналитически, всегда однозначно)
- Inverse Kinematics (IK): pose end-effector → θ1...θn (численно, не всегда существует, может быть неоднозначна)
  - KDL Solver: Jacobian-based, может попасть в локальный минимум
  - TRAC-IK: комбинирует несколько методов, быстрее и надёжнее KDL
  - bio-IK: эволюционный, поддерживает дополнительные цели (secondary objectives)

### Планирование в MoveIt
- Установка целей: `setJointValueTarget()`, `setPoseTarget()`, `setPositionTarget()`
- `plan()`: возвращает `MoveGroupInterface::Plan` — траектория + метаданные
- `execute()` vs `move()`: execute асинхронный, move блокирующий
- `computeCartesianPath()`: картезианские waypoints → joint trajectory (fraction — доля пути)
- Constraints API:
  - JointConstraint: ограничить сустав в диапазоне
  - PositionConstraint: ограничить позицию end-effector в объёме
  - OrientationConstraint: ограничить ориентацию
  - PathConstraint vs GoalConstraint: на всём пути vs только в цели

### Planning Scene Management
- `addCollisionObject()`: box, sphere, cylinder, mesh (STL)
- `attachObject()` / `detachObject()`: gripper захватил объект → он движется вместе с роботом
- `applyPlanningScene()`: синхронизация с MoveIt
- Octomap Plugin: автоматически строит препятствия из point cloud sensor

### OMPL в MoveIt
- RRTConnect: по умолчанию, хорош для большинства задач
- RRT*: когда важно качество траектории
- PRM: при многих запросах для одной планировочной задачи
- LazyPRM: для разреженных препятствий
- Настройка через `planning_plugin_config.yaml`

### Post-processing траектории
- `computeTimeParameterization`: добавление скоростей/ускорений
- IterativeParabolicTimeParameterization (IPTP): стандартный метод
- Time-Optimal Path Parameterization (TOPP-RA): учитывает динамику — быстрее и реалистичнее
- Trajectory smoothing: Ruckig jerk-limited profile

## Часть VI. Behavior Trees и конечные автоматы (FSM)

### Finite State Machine (FSM)
- Состояния (states) + переходы (transitions) + условия (guards) + действия (actions)
- Простота: идеально для простых задач с малым числом состояний
- Проблема: «state explosion» — с ростом сложности задачи граф становится нечитаемым
- Иерархические FSM (HFSM): состояния-суперсостояния, вложенность — снижает сложность
- Инструменты: SMACH (ROS 1 legacy), FlexBE (ROS 2), BehaviorTree.CPP

### Behavior Trees (BT) — деревья поведений
- Структура: Control nodes (Sequence, Fallback, Parallel) + Execution nodes (Action, Condition)
- **Sequence** (→): выполняет дочерние слева направо, останавливается при первом FAILURE
- **Fallback** (?): выполняет дочерние слева направо, останавливается при первом SUCCESS
- **Parallel**: запускает всех дочерних одновременно
- **Action**: вызывает внешний код (ROS action / service / функция)
- **Condition**: проверяет условие → SUCCESS или FAILURE
- Статусы: RUNNING, SUCCESS, FAILURE

```
Пример: NavigateToPose BT
Root [Sequence]
  ├── IsRobotLocalized? [Condition]
  ├── ComputePathToPose [Action]
  ├── FollowPath [Action]
  └── [Fallback]
        ├── IsPathClear? [Condition]
        └── RecoveryBehavior [Action]
```

### Преимущества BT над FSM
- Модульность: поддерево можно переиспользовать
- Реактивность: тикование (ticking) позволяет прерывать действия
- Читаемость: дерево понятнее большого графа переходов
- Тестируемость: каждый узел тестируется изолированно

### BehaviorTree.CPP
- C++ библиотека, интеграция с ROS 2 (nav2_bt_navigator использует её)
- XML описание дерева: загрузка без перекомпиляции
- Порт BlackBoard: разделяемая память между узлами дерева
- Кастомные узлы: `registerNodeType<MyAction>()`
- Groot2: GUI редактор и мониторинг деревьев в реальном времени

### Practical Design Patterns
- Retry decorator: повторить N раз при FAILURE
- Timeout decorator: принудительное завершение при превышении времени
- Inverter decorator: инвертировать SUCCESS/FAILURE
- SubTree: инкапсуляция сложного подзадачи

## Часть VII. Multi-robot Coordination и формирование

### Проблема координации нескольких роботов
- Multi-Robot Path Planning (MRPP): найти пути для N роботов без столкновений
- Complexity: MRPP на сетке — PSPACE-hard в общем случае
- Coupled vs decoupled planning: совместное C-space vs независимые планы + разрешение конфликтов

### Decoupled подходы
- Priority-based planning: роботы планируют в порядке приоритета, ранее запланированные — «препятствия»
  - Простой, быстрый, но не оптимален и неполон
- Velocity Obstacles (VO): робот вычисляет зоны скоростей, ведущих к столкновению
  - Reciprocal Velocity Obstacles (RVO): взаимная ответственность за уклонение
  - ORCA (Optimal Reciprocal Collision Avoidance): гарантированное уклонение

### Coupled подходы
- Conflict-Based Search (CBS): оптимальный multi-agent pathfinding
  - High-level: обнаруживает конфликты между парами агентов
  - Low-level: перепланирует для конкретного агента с ограничением-конфликтом
  - ECBS: Enhanced CBS с ε-оптимальностью — быстрее
- M* : оптимальный, учитывает структуру C-space

### Формирование (Swarm / Formation)
- Leader-follower: ведущий + последователи поддерживают relative offset
- Virtual structure: роботы — части жёсткой виртуальной структуры
- Behavior-based formation: локальные правила → глобальное поведение (Boids)
- Formation control в ROS 2: каждый робот — отдельный Nav2 стек, координация через топики

### Task Allocation
- Hungarian algorithm: оптимальное назначение N роботов на N задач (bipartite matching)
- Auction-based: каждый робот «торгуется» за задачи — распределённый подход
- Возможности для масштабирования: contract-net protocol

## Часть VIII. Планирование под неопределённостью: POMDP, Belief Space

### Источники неопределённости в робототехнике
- Сенсорный шум: GPS точность ±3м, LiDAR погрешность, camera depth
- Актуационный шум: колёса проскальзывают, сервоприводы имеют люфт
- Неполное знание окружения: невидимые области, динамические объекты

### MDP (Markov Decision Process)
- Кортеж (S, A, T, R, γ): состояния, действия, переходы, награды, коэффициент дисконтирования
- T(s, a, s'): вероятность перехода из s в s' при действии a
- R(s, a): награда за выполнение действия a в состоянии s
- Цель: найти политику π: S → A максимизирующую ожидаемую суммарную награду
- Value Iteration: V(s) = max_a [R(s,a) + γ * Σ T(s,a,s') * V(s')]
- Policy Iteration: чередование оценки и улучшения политики

### POMDP (Partially Observable MDP)
- Реальность: робот не знает своего точного состояния s ∈ S
- Observation model: O(o | s', a) — вероятность наблюдения o из состояния s'
- Belief state b(s): распределение вероятностей над возможными состояниями
  - Обновление belief: b'(s') ∝ O(o | s', a) * Σ T(s, a, s') * b(s)
- Policy: π: B → A (действие зависит от belief, не от состояния!)
- Решение POMDP: PBVI, SARSOP — точные методы; POMCP — монте-карло для больших пространств
- Применение: захват объектов при частичной видимости, поиск и спасение

### Belief Space Planning
- Цель: найти траекторию в belief space от b0 к b_goal
- Информационная ценность действий: некоторые действия снижают неопределённость
- iLQG (iterative Linear Quadratic Gaussian): линеаризация + gaussian belief
- Практика: Robot Localization (AMCL в Nav2): particle filter — дискретная аппроксимация belief

### Robust Planning
- Minimax: план для наихудшего сценария — консервативный
- Chance Constraints: P(столкновение) < ε — вероятностные ограничения
- Scenario-based optimization: оптимизируй для набора сценариев

# 3. ИСТОРИЧЕСКИЙ КОНТЕКСТ

## Ключевые вехи

> **1979 — Lozano-Perez & Wesley:** Первая формализация C-space для планирования движения.
> Заложили математический фундамент всей области.

> **1996 — LaValle (RRT):** Революционный алгоритм для high-dimensional spaces.
> RRT стал стандартом для планирования манипуляторов. Использован в тысячах роботов.

> **1998 — Kavraki et al. (PRM):** Probabilistic Roadmap — multi-query планирование.
> OMPL сегодня содержит оба алгоритма как базовые.

> **2002 — Koenig & Likhachev (D* Lite):** Динамическое перепланирование.
> NASA Spirit/Opportunity Mars Rover использовал D* для навигации на Марсе (2004-2010).

> **2005 — Colledanchise & Ögren:** Behavior Trees из игровой индустрии в робототехнику.
> Halo (2001) использовал BT для AI врагов. Сейчас — стандарт в автономных системах.

> **2011 — CBS (Conflict-Based Search):** Sharon et al. — оптимальный multi-agent pathfinding.
> Amazon Robotics использует варианты CBS для тысяч роботов на складах.

## Форматы проверки

1. **Алгоритмический разбор** — объясни работу на конкретном примере

4. **Системное проектирование** — спроектируй систему планирования для задачи
5. **Отладка** — дан сломанный планировщик, найди причину

### Алгоритмический разбор

```
Дано: 2D сетка 5x5. Робот в (0,0). Цель в (4,4).
Препятствия: (1,1), (2,1), (3,1), (1,2), (1,3).
Эвристика: Manhattan distance.

Прогони A* вручную:
1. Покажи первые 5 итераций: что в OPEN, что расширяем
2. Какой путь найдём? Оптимален ли он? Докажи.
3. Если заблокировать (4,3): что изменится в поиске?
```

### Сравнительный анализ

```
Сценарий: Роботизированная рука (7-DOF) в лаборатории.
Задача: захват объектов на конвейере (~10 объектов/минуту).
Препятствия: статичны, кроме самого объекта.

Вопросы:
1. Почему A* не применим напрямую? (подсказка: размерность)
2. PRM или RRT для этой задачи? Аргументируй.
3. Если объект иногда закрыт другим объектом — как это влияет на выбор?
4. Как учесть ограничение: ориентация захвата должна быть сверху вниз?

```

```

- Коридоры шириной 2 метра
```
